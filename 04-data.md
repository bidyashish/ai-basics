# 04 · Data — Collect, Process, and Prepare

> **TL;DR** In 2026, the easiest path to a strong small model is to start from **FineWeb-Edu** or **DCLM-Baseline** (both open, both pre-cleaned), mix in code (StarCoder2 / The-Stack-v2), math (FineMath), and synthetic instruction data (Cosmopedia / OpenHermes / Tülu 3). Then dedupe, filter for quality, and shard into tokenized binary files that a `DataLoader` can stream. **The quality of your tokens beats the quantity of your tokens.**

## 1. Why data is the model

Modern LLMs are mostly defined by their data. Two teams using the same architecture and the same compute can produce wildly different models because one had cleaner data. This was less obvious in 2020 when the architecture was the bottleneck. In 2026, **architecture is essentially solved for the small-model regime; data is the lever.**

Three data axes matter:

1. **Quality** — is the text correct, well-written, informative?
2. **Diversity** — does it cover many topics, styles, languages, modalities?
3. **Quantity** — measured in tokens, not GB. ~4 chars/token for English.

Better data lets you train smaller models for fewer steps and still beat bigger models on benchmarks.

---

## 2. The 2026 open-data landscape

You almost never crawl the web yourself. Use the curated open datasets from HuggingFace.

### General web text

- **FineWeb** (~15T tokens) — HuggingFace's clean Common Crawl. Default base.
- **FineWeb-Edu** (~1.3T tokens) — FineWeb filtered by an educational-content classifier. Punches *way* above its weight; small models trained on it match models trained on 5× more raw FineWeb.
- **DCLM-Baseline** (~4T tokens) — MIT/Apple's release; competitive with FineWeb-Edu, slightly different filtering style.
- **The Pile** — older but still useful for reproducibility and ablations.

### Code

- **StarCoder2 / The-Stack-v2** — permissively-licensed code from GitHub. ~900B tokens.
- **CodeContests, OpenCodeReasoning** — competition-style code with reasoning traces.

### Math and reasoning

- **FineMath** — math web pages filtered by a classifier. Surprisingly hard to find at scale.
- **OpenWebMath**, **InfiMM-WebMath**, **MathPile**.
- **Synthetic math** like NuminaMath, OpenR1-Math, AceMath.

### Multilingual

- **HPLT v2**, **CulturaX**, **MADLAD-400** — non-English crawls.
- For a strong English model, you still want 5-15% non-English to keep "reasoning patterns" diverse.

### Instruction / chat (for SFT, see chapter 14)

- **Tülu 3 SFT mix**, **Llama-Nemotron post-training**, **Open-Orca-2**, **OpenHermes-2.5**.
- **Cosmopedia v2** — fully synthetic, extremely clean instruction data generated by Mixtral & Llama.
- **UltraFeedback / HelpSteer3** — for preference optimization.

### Multimodal (if relevant)

- **OBELICS** (interleaved image-text), **DataComp**, **LAION-5B-en-aesthetic-v2**.

## 3. Sourcing your own data (when you must)

If you want data the open releases don't have (a niche domain, a private corpus), the pipeline is:

1. **Crawl** — `commoncrawl.org` is the standard source. Use **WARC** files. `warcio` parses them.
2. **Extract text** — `trafilatura` or `resiliparse` strips boilerplate from HTML cleanly. **Boilerplate is poison;** menus, footers, and copyright notices repeated billions of times will tank your model.
3. **Detect language** — `fasttext-langdetect` is fast and accurate.
4. **Filter** — short docs, gibberish, repetitive boilerplate.
5. **Deduplicate** — across documents and within documents.
6. **Quality-classify** — keep the educational, drop the spam.

Realistic numbers: a clean pipeline keeps **5-15%** of raw Common Crawl HTML as final training data. The rest is junk.

---

## 4. Cleaning rules of thumb

Bad data crushes models. Some heuristics that work:

- **Length filter**: drop docs < 200 chars or > 100k chars (per page).
- **Mean word length**: 3-10 chars typical English. Outside is gibberish.
- **Symbol/word ratio**: > 0.1 is suspicious (tables, code dumps in prose data).
- **Bullet/ellipsis ratio**: > 90% bullets is usually a navigation page.
- **Repeated n-gram ratio**: top 5 4-grams covering > 20% of text → spam/SEO.
- **Profanity / NSFW classifiers**: subjective; depends on your goal.
- **PII redaction**: emails, phone numbers, credit cards if you're shipping the data.

Gopher-style filters and the C4-clean rules are open-source recipes you can reuse.

## 5. Deduplication: cheap, big win

LLMs memorize. If you train on the same sentence 100×, the model wastes capacity remembering it instead of learning. **Deduplication often gives a free perplexity win equivalent to training on 2× more compute.**

Two flavors:

### Exact dedup
Hash each document (SHA-256 or just MD5). Drop duplicates. Easy.

### Near-duplicate dedup (MinHash + LSH)
Documents that are 90% the same but not identical (e.g., the same article reprinted with a different header). The standard tool is **MinHash with locality-sensitive hashing**:

1. Tokenize each doc into n-grams (5-grams of words).
2. Compute MinHash signature (e.g., 256 random hashes, store the min hash from each over the n-grams).
3. Bucket documents by overlapping signature bands; check candidates for true Jaccard similarity > 0.8.
4. Drop duplicates within each bucket.

Tools: **`datasketch`** (Python), **`text-dedup`** (HuggingFace), **`spark-rapids-ml`** for large scale.

For a few TB of text, near-dedup runs in hours on a beefy CPU machine.

### In-document dedup

Some docs paste the same paragraph twice. A simple `set()` of paragraph hashes per doc helps.

## 6. Quality filtering with classifiers

Heuristics catch obvious junk; **classifiers catch the boring-but-low-information stuff**.

The 2026 standard recipe:

1. Take a small **gold set** (~100k docs) labeled high/low quality. Sources:
   - High: Wikipedia, books, arXiv abstracts, academic papers.
   - Low: random Common Crawl sample (mostly forums, ads, SEO).
2. Train a fast classifier (logistic regression on hashed features, or a small fastText / DistilBERT model) to predict "is this educational/high-quality?"
3. Score every document, keep top X%.

This is exactly how **FineWeb-Edu** was built: train a classifier on Llama-3-70B-labeled "educational value" scores, then filter FineWeb to docs with score ≥ 3 out of 5.

DCLM did similar with a different model. Both are now standard tools you can either use directly or replicate.

```python
# rough sketch with fasttext
import fasttext
clf = fasttext.train_supervised('quality_train.txt',
                                lr=1.0, epoch=5, wordNgrams=2, minCount=3,
                                dim=64, loss='hs')
clf.save_model('quality.bin')

# at scale
def keep(doc):
    label, prob = clf.predict(doc.replace('\n', ' '))
    return label[0] == '__label__high' and prob[0] > 0.5
```

## 7. Decontamination

Your model will be evaluated on benchmarks (MMLU, GSM8K, HumanEval, etc.). If those exact questions are in your training data, you're cheating yourself. Run **decontamination**:

1. Hash all benchmark prompts (n-grams of length 13 is standard).
2. Search training data for any doc that contains a benchmark n-gram.
3. Remove (or at least flag) such docs.

This is standard practice. Tools: HuggingFace's `decontaminate` script, `datatrove`.

---

## 8. Data mixing

Your final pretraining corpus is a **mixture**. Common ratios for a 2026 small model:

| Source | Share |
|--------|-------|
| Clean general web (FineWeb-Edu / DCLM) | 60-70% |
| Code (StarCoder2 filtered) | 15-25% |
| Math (FineMath, OpenWebMath) | 5-10% |
| Books / academic / Wikipedia | 5-10% |
| Multilingual (small) | 0-10% |

Higher code share helps reasoning even on non-code benchmarks (papers like "Code Pre-training Improves Reasoning" show this).

You can also do **curriculum learning**: noisier general data first, then later epochs on cleaner / harder data (math, instruct-style). Very common in 2026 — the last ~10% of pretraining tokens is often called the "anneal phase" and uses heavily filtered data.

## 9. Tokenizing and sharding for training

Once data is cleaned, the training loop wants **fixed-size sequences of token IDs, fast to read, cheaply shuffled**. The standard format:

1. **Tokenize** every document with your model's tokenizer (chapter 6).
2. **Concatenate** with end-of-document tokens between them: `[BOS] doc1 tokens [EOS] doc2 tokens [EOS] ...`
3. **Pack into fixed-length sequences** of `block_size` (e.g., 4096). No padding, no waste.
4. **Save as binary shards** of e.g. 1B tokens each. uint16 or uint32 depending on vocab size.

Fixed-size packing is critical for throughput — the alternative (variable-length with padding) wastes compute.

```python
# minimal tokenize-and-shard sketch
import numpy as np
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained('meta-llama/Llama-3.2-1B')
shard_size = 1_000_000_000
buf = []
shard_id = 0
def flush():
    global shard_id, buf
    arr = np.array(buf, dtype=np.uint32)
    np.save(f'shard_{shard_id:05d}.npy', arr)
    shard_id += 1
    buf = []

for doc in stream_docs():           # your iterator over cleaned docs
    ids = tok.encode(doc) + [tok.eos_token_id]
    buf.extend(ids)
    if len(buf) >= shard_size:
        flush()
flush()
```

For a real run, use **`datatrove`** (HuggingFace) or **`mosaic-streaming`** which handle distributed processing, sharding, resumable runs, and S3 streaming.

## 10. Streaming at training time

You don't load 15T tokens into RAM. You stream them.

The two mainstream options:

- **`mosaicml-streaming`** — designed for huge datasets on S3/GCS. Each rank reads disjoint shards. Excellent shuffling. Works with PyTorch `DataLoader`.
- **`HuggingFace datasets` with `streaming=True`** — easier to start, fine up to mid-scale.

A minimal streaming dataset:

```python
from torch.utils.data import IterableDataset, DataLoader
import numpy as np, glob, random

class ShardedDataset(IterableDataset):
    def __init__(self, shard_glob, block_size=4096):
        self.shards = sorted(glob.glob(shard_glob))
        self.block_size = block_size

    def __iter__(self):
        random.shuffle(self.shards)
        for shard in self.shards:
            data = np.load(shard, mmap_mode='r')
            n = (len(data) - 1) // self.block_size
            order = np.random.permutation(n)
            for i in order:
                start = i * self.block_size
                x = data[start : start + self.block_size]
                y = data[start + 1 : start + 1 + self.block_size]
                yield x.astype(np.int64), y.astype(np.int64)

loader = DataLoader(ShardedDataset('shards/*.npy'),
                    batch_size=8, num_workers=4, pin_memory=True)
```

`mmap_mode='r'` lets you stream a 1B-token shard without loading it into RAM. The OS pages it in.

## 11. Data is the experiment, not a fixed input

Once your training pipeline works, **data ablations** are how you actually improve:

- Add 5% more code → did MMLU go up? did GSM8K?
- Replace 100B FineWeb with 100B FineWeb-Edu → did everything go up?
- Drop multilingual → did English perplexity improve? (Often yes by 1-2%, but you lose multilingual ability.)
- Anneal phase with curated high-quality math → does GSM8K spike?

Set up ablations early. Train a 100M-300M token "scout" model on each variant for a few hours, compare. Many papers in 2025-2026 are *just* data ablations.

---

## 12. A 2026 cheat sheet

- Don't crawl. **Use FineWeb-Edu or DCLM-Baseline** as your base.
- Mix in StarCoder2-filtered code, FineMath, Cosmopedia for synthetic.
- Run **MinHash near-dedup** on your full mix.
- Run a **classifier-based quality filter** if you're not already using a curated set.
- **Decontaminate** against benchmarks you'll evaluate on.
- Tokenize once with your chosen tokenizer; shard into 1B-token uint32 binary files.
- Stream with `mosaicml-streaming` or a custom `IterableDataset`.
- Use the last ~10% of training as an **anneal phase** on extra-clean data.
- Track data lineage like code: which model was trained on which data version. You will need to know this.
